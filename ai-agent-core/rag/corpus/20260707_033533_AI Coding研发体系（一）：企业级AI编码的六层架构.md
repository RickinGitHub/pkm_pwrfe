# AI Coding研发体系（一）：企业级AI编码的六层架构

AI Coding 研发体系｜第一篇

这一篇先用一张总架构图说明：企业做 AI Coding，不是买几个工具，而是要补基础设施、Agent、流程、组织、指标和治理六层能力。后面会继续逐层拆解，适合连起来看。

很多团队推进 AI Coding，第一步都是买许可证。

Cursor、Copilot、Codex、Claude Code 很快被装到开发机上，知识库也接了，编码规范也喂了。短期看，个人效率确实会变快。

但到了企业级代码库里，问题很快换了一批：谁给 AI 权限，谁验证它的改动，Token 成本怎么算，测试失败谁负责，AI 生成的代码能不能直接进主干。

**这时 AI Coding 就不再是工具采购问题，而是研发体系设计问题。**

如果只从工具角度看，企业会不断比较 Cursor、Copilot、Codex、Claude Code 谁更强。但从架构角度看，真正要回答的是另一组问题：AI 在什么环境里工作，承担什么角色，进入哪段流程，由谁监督，用什么指标评价，又被哪些治理规则约束。

所以我更倾向于把企业级 AI Coding 看成一套研发操作系统，而不是一组工具清单。它至少包含六层：基础设施、Agent 执行、研发流程、组织能力、评价指标和治理体系。下面这张图会作为整个系列的总图母版，后续每篇都会回到它，再拆其中一层或一个专题。

架构图：后续每篇会沿着这张图拆一层或一个专题。

这张图不是一张采购清单，而是一张责任分布图。越往下，越接近模型、工具、代码库和系统集成；越往上，越接近组织、指标、权限、质量和责任。企业做 AI Coding，难点通常不是某一层完全没有，而是层与层之间没有被设计成一条生产链路。

放到真实研发现场里看，一个需求不会只经历“AI 写代码”这一步。它要先被拆成明确任务，再找到相关代码、规范和历史实现；Agent 生成修改后，还要跑测试、过 Review、补安全检查，最后进入 PR、发布审批和回滚预案。六层架构讲的，就是这条链路里每个环节该由什么能力接住。

| 层级 | 架构问题 | 缺失后的表现 |
|---|---|---||
| 基础设施 | AI 运行在什么资源和工具上 | 只能个人试用，难进企业环境 |
| 执行层 | 哪些 Agent 承担哪些任务 | 单个 Agent 很忙，但协作混乱 |
| 流程层 | AI 产出如何进入研发链路 | 生成很快，返工和验证更重 |
| 组织层 | 人如何监督、编排和验收 AI | 工程师还按旧角色工作 |
| 评价层 | AI 是否真的改变交付结果 | 只剩采纳率和代码占比 |
| 治理层 | 权限、成本、质量和责任谁兜底 | 自动化越强，风险越难追踪 |

后续每篇文章都会从这张图出发：先说明自己处在哪一层，再解释它和上下层如何连接。这样看 AI Coding，就不会只停在某个工具、某个 Prompt 或某次实测上，而是回到企业研发系统本身。

### 
最容易被看见的是基础设施层。

模型选 GPT 系列、Claude、Gemini，还是通义千问；工具选 Codex、Cursor、Claude Code，还是 Roo Code；知识层接代码库、设计文档、API 文档和历史 Issue；再往外接 Git、Jira、Confluence、数据库、测试平台和 CI/CD。

这些都重要，但它们只回答一个问题：AI 能不能接进来。

企业最常见的误判，是把“接进来”当成“跑起来”。买了许可证、部署了模型、连了知识库，确实完成了一步，但这一步更像铺路，不是交付本身。

**买工具只能让 AI 进门，不能保证它进入研发生产线。**

| 企业动作 | 解决的问题 | 还没解决的问题 |
|---|---|---||
| 买工具 | 让工程师能用 AI | AI 如何进入流程 |
| 接知识库 | 让 AI 看见资料 | 哪些上下文该看 |
| 接 CI/CD | 让 AI 能触发验证 | 失败以后谁判断 |
| 开权限 | 让 AI 能执行任务 | 越界以后谁兜底 |

### 
AI Coding 真正变复杂，是从执行层开始的。

个人使用时，一个 Coding Agent 就够了：写代码、修 Bug、补测试、解释报错。企业研发不是这样。一个需求从提出到上线，中间会经过产品拆解、架构设计、代码实现、Review、测试、发布和回滚预案。

所以 Agent 也会分工。PM Agent 拆需求，Architect Agent 做方案比较，Coding Agent 改代码，Review Agent 查规范和安全，QA Agent 设计测试，Ops Agent 看发布和故障处理。

但分工本身还不够。更关键的是流程层：这些 Agent 什么时候介入，读什么上下文，产出什么结果，谁来验收，失败以后回到哪一步。

传统流程是需求、设计、开发、测试、上线。AI 进入以后，中间会多出 AI 规划、AI 编码、AI 测试、人工验证和反馈修复。看起来只是多了几个步骤，实际是在重排软件研发的责任链。

**单个 Agent 再强，如果流程仍然靠人手动搬运上下文，效率会被很快吃掉。**

| 旧问题 | 工具视角 | 体系视角 |
|---|---|---||
| AI 会不会写代码 | 看模型和 IDE | 看需求、上下文和验证是否打通 |
| AI 能不能修 Bug | 看单次生成质量 | 看日志、测试、回归和 Review 闭环 |
| AI 是否省人力 | 看代码占比 | 看交付周期、返工率和验证成本 |
| AI 是否安全 | 看工具权限开关 | 看审计、回滚、职责和发布门禁 |

### 
组织能力层，是很多企业最容易低估的一层。

过去评价工程师，主要看编码、调试、设计和交付。AI 辅助编码之后，又多了 Prompt、工具使用和自动补全能力。但这些还只是早期能力。

真正的变化会出现在监督式工程师身上。

这个角色不一定少写代码，但工作重心会明显迁移：把需求表达清楚，把上下文组织好，把验收标准说完整，把 AI 的中间结果拉回正确方向，把最终风险拦下来。

再往上，AI Tech Lead 要管理多个 Agent 的任务编排、架构边界和技术决策；AI Engineering Manager 要设计指标、治理和 ROI 口径。

**当 AI 能写更多代码，人类工程师的价值不会消失，而是转移到意图、上下文、验证和责任。**

### 
AI Coding 很容易被几个漂亮数字带偏。

比如工具覆盖率、DAU、WAU、AI 代码占比、Prompt 次数。这些数字不是没用，它们能说明采纳情况和使用深度。但它们不能直接证明研发体系变好了。

企业真正关心的应该往后看：需求交付周期有没有缩短，PR 周期有没有改善，Bug 率有没有上升，Review 打回率有没有变化，人均交付和成本节省是否能算清楚。

到了 Agent 时代，还要加一组新指标：一次成功率、平均反馈轮次、AI 返工率、验证成本、Intent Efficiency。它们问的不是 AI 用得多不多，而是人机协作有没有真正变顺。

| 指标 | 看起来在问 | 实际在问 |
|---|---|---||
| AI 代码占比 | AI 写了多少 | 这些代码是否真的进入交付 |
| 一次成功率 | AI 是否靠谱 | 需求、上下文和验证是否足够清楚 |
| 平均反馈轮次 | 人要改几次 | 人机协作成本是否变高 |
| AI 返工率 | AI 有没有写错 | 任务边界和验收标准是否稳定 |
| 验证成本 | 测试花多久 | 省下的编码时间是否被验证吃掉 |

**AI 写了多少代码不是终点，企业要问的是交付、质量和责任有没有一起变好。**

### 
治理层听起来最不性感，但它会决定企业级 AI Coding 的上限。

模型版本、Prompt 版本、Agent 版本要能追踪；AI 代码来源、版权和归属要能说明；敏感代码、数据泄露和供应链风险要能拦住；Token 消耗、调用成本和预算归因要能算；生成代码质量、测试覆盖率和发布质量要能看。

以后还会多出一个新领域：Agent 治理。

Coding Agent 可以改代码，但不应该直接上线。更稳的流程应该是：Coding Agent 产出修改，Review Agent 检查规范和安全，QA Agent 设计和执行验证，最后经过人工审批或发布门禁。

**个人可以容忍一次 AI 改错文件，企业不能容忍一套没人负责的自动化研发链路。**

AI Coding 的下一阶段，不是继续比较哪个工具更聪明，而是看企业能不能把基础设施、Agent、流程、组织、指标和治理连成一套可运行系统。

对个人开发者来说，工具能力决定体验的一部分。

对企业来说，真正的分水岭会变成：谁能把 AI 写代码这件事，变成可验证、可度量、可治理的研发生产方式。

我会按这张图继续拆流程层、组织能力层、评价层和治理层，尤其是监督式工程、Context Engineering 和 Verification Engineering。AI Coding 不会只改变写代码的人，它会改掉一整套研发系统的运行方式。
