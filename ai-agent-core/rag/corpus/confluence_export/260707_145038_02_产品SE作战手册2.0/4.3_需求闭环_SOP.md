# 4.3 需求闭环 SOP

> pageId: 678927551 | 导出时间: 2026-07-07T14:50:56.146699

# **SOP简介：**

**文档主要内容：**规范需求从提出到关闭的全流程，确保需求可追溯、可执行、有结果

**文档适用角色：**大产品SE（机芯软件负责人）& 小产品SE（派生软件负责人）

**适用项目阶段：SR2 / SR3 / SR4 / 量产前**

**环境依赖：**gerrit 代码上传、下载权限

**相关内容链接：**

[深圳TCL新技术有限公司代码管理规范](https://confluence.tclking.com/pages/viewpage.action?pageId=181198800)

智屏侧的block问题定义

## 1. 目的

通过规范需求从 需求**范围评审 → 需求开发 → 量产运作 **的完整流程，确保：

- 需求目标清晰
- 导入和交付时间明确
- 涉及模块、owner明确

降低沟通成本，避免需求遗漏、理解偏差导致需求不能按时交付的问题。

## 2. 需求闭环流程

### 2.1 需求准入确认

需求进入分支前，必须确认以下信息：

1. 通过上车质量门禁
2. 有完整的上车评审表且需求上车评审通过
3. 影响范围（功能 /机芯/模块）明确及feature隔离

### 2.2 量产**运作**

1. 确认需求范围：在新机芯立项、版本火车发车、班车需求评审时，需要确认需求的范围，包括涉及机芯、需求数量
2. 需求导入：

- 与需求大A再次确认需求涉及模块（系统、应用、soc）及所有修改点，避免修改遗漏，保证一个版本合入所有修改点
3. 安排开发将修改导入分支，并按计划发布版本进行测试

- 测试进展跟进：

1. 告知VPM哪个版本导入了需求，并在晨会上与VPM确认测试进展，及时处理BLOCK测试的测试项
2. 日常关注新增问题，并通过SPM邮件发项目日报以及线上与开发过问题相方式，推动开发处理问题，在下一个版本发布前，尽量多的处理问题（版本释放前，BLOCK问题必须清零）

- 版本发布：

1. 发版前，检查jira问题的状态，转release的问题需要将修改合入对应分支，避免因某些重大修改修改未合入导致版本重发
2. 在过程版本测试完成后，由SPM组织产品SE、VPM、质量SE组织问题评审，转SCCB的问题需让开发提前备注五维分析，识别出其中的BLOCK问题

- 版本交付：软件达到释放标准后可拉起评审，评审通过后软件方可释放

### 2.3 需求闭环判定（核心）

需求闭环必须同时满足以下条件：

1. 需求已完整实现
2. 所带需求的版本已释放
3. 未闭环问题必须明确：

- 

- 风险点
- 影响范围
- 后续解决计划

### 2.4 需求复盘与改进

- 在后续的回顾会议中，参与需求完成率、延期原因、质量问题进行分析，不断优化运作流程，提高需求闭环效率

### 2.5 需求价值验收

需求验收方式由灵悉OS团队进行确认，在需求上线一定时间后可以与验收方确认验收的结果

## 3. 关键管控点

| 管控点 | 要求 |
| --- | --- |
| 分支准入 | 无验收标准需求禁止进入 |
| 合入检查 | 需求必须可追溯到 JIRA |
| 验证结论 | 无验证结论不得关闭需求 |
| 发布前确认 | 未闭环需求需显式风险说明 |
