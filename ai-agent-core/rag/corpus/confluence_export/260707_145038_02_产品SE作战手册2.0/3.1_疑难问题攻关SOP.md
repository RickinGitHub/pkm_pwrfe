# **疑难问题攻关SOP**

**一、疑难问题攻关范围：**

1、开发已投入分析但短期无有效解决方案的问题

2、跨模块跨专业，无人主导分析的问题

3、模块owner提出需要协助的问题

**二、攻关步骤
**

1、产品SE了解问题现状，比如：机型信息、问题阻塞在哪，是否有现象和日志，sqa是否模拟复现

2、把相关模块owner和SE、模块SE、SQA拉群沟通（如果是工厂问题需要把AOE和DFM 拉进去群、如果是售后问题需要把售后服务代码拉群、如果是TQC问题需要把TQC拉进群）；如果问题模块明确可以根据模块责任田确认owner，如果问题模块不明确先从问题发生场景(上层报错的应用)、log相关报错信息、操作涉及的模块出发， 先抓上层应用的owner再逐步拉通中间件和底层的woner对齐，大体确认问题相关onwer（宁可错抓，不可漏网一人）。

3、如果问题缺少日志或现象视频：

（1）拉通模块owner、模块SE、SQA、问题反馈接口人（工厂问题AOE、售后问题售后服务代表等）对齐模拟方法和抓bug方式，SQA进行模拟复现

（2）联系售后服务代表（售后问题）、AOE（工厂问题）、TQC 远程协助抓取log和提供现象视频，如有必要安排模块owner现场debug

4、如果问题已经有日志或现象视频：

（1）SQA对比竞品或历史版本是否有相同问题

（2）模块owner根据现有log分析，如果分析需要其他模块或SOC协助，产品SE组织拉通相关模块人员进行讨论下一步实验方案、问题风险点、问题是否有规避措施等，实时在群里或会议同步进展

（3）微信和T信群及时同步实验效果，每天会议对齐问题汇总当前进展以及制定下一步策略；产品SE需深度参与方案讨论，了解方案的设计逻辑和风险，站在产品的角度发表方案建议，群策群力确保方案不跑偏，问题能得到解决且方案对整体项目影响最小。

（3）产品SE根据风险点评估对整个项目的影响（比如是否会阻塞正常软件释放，是否需要叫停生产或ota的版本），如果问题无法满足项目释放要求，及时上升风险到部门长，并拉通整个项目组制定新版本计划为项目保驾护航

（4）如果问题有解决措施，需先keji构建临时版本给SQA验证，如果是工厂问题同步给AOE验证、如果是售后问题同步给售后服务代表给客户验证，如果是tqc问题同步给tqc验证（如果问题只有特定区域才能复现的，必须经过当地验证），临时版本验证pass后再构建正式版本提测

（5）如果问题问题卡壳，经过大家努力还是没有解决方案需上升SEG进行攻关（参考SEG攻关流程），如果SEG决策也没有技术解决方案，需要及时同步风险给产品线，从产品设计考虑规避方案。

备注：行动项实时同步，及时拉会对齐进展，避免问题卡壳停滞不前，直到问题解决
