> pageId: 583197323 | 导出时间: 2026-07-07T14:51:17.479723

# **SOP简介：**

**文档主要内容：**在软件释放前一定要提测TQC，TQC是软件释放重要环节，需要无block问题。特别EU、JP、阿根廷、巴西、NA需要TQC 释放后才能最终软件释放

**文档适用角色：**VPM，SPM，SE，TQC

**适用项目阶段：**SR4 SR5

**环境依赖：**

**相关内容链接：**

## 核心红线

- 软件释放前必须完成 TQC 提测
- TQC 阶段必须无 block 问题
- EU、JP、AR、BR、NA 等重点区域，必须在 TQC 释放后才能进行最终软件释放

## 一句话理解

TQC 是重点区域软件最终释放前的关键质量门禁，不是可选动作。

# 1. 流程总览

TQC 场测风险管理整体流程如下：

1. 样机准备
2. 测试计划沟通
3. 版本提测
4. 问题处理
5. 问题评审
6. TQC 结论输出
7. 软件最终释放

# 2. 角色职责

| 角色 | 职责说明 |
| --- | --- |
| VPM | 负责同步 TQC、推动问题复现与风险协调 |
| SPM | 负责测试计划沟通、版本提测、组织问题评审和整体推进 |
| SE | 负责问题支持、区域沟通和质量协同 |
| TQC | 负责场测执行、问题反馈、协助复现和测试结论输出 |
| 开发 Owner | 负责问题分析、定位和修复 |
| 开发代表 | 负责测试样机寄送安排和开发侧协调 |

# 3. 测试计划管理

不同区域的 TQC 周期和沟通方式不同，必须在项目排期阶段提前规划。

## 3.1 区域测试周期要求

| 区域 | 测试周期 / 管理要求 |
| --- | --- |
| EU / AR | 提前 1 周沟通测试计划 |
| BR | 提前 1 周提测 |
| JP | 预留 2 周测试时间 |
| NA | 根据修改点预留 3～7 天 |

## 3.2 提测前提

提测前应满足以下条件：

- 终版软件已发布
- SQA 已测试 1～2 天且无严重问题
- 已完成区域测试计划沟通

## 3.3 管理要求

- TQC 周期必须前置到项目排期中
- 不允许等到最终发布前再临时协调
- 重点区域必须预留足够验证时间

# 4. 样机准备

## 4.1 基本要求

TQC 测试样机需提前准备，避免影响测试排期和版本释放。

## 4.2 职责分工

- **SPM：**负责与开发代表沟通样机需求
- **开发代表：**负责安排样机寄送

## 4.3 管理要点

- 样机准备不及时，会直接影响 TQC 开始时间
- 样机寄送节点需与测试计划同步确认

# 6. 问题处理机制

## 6.1 总原则

TQC 提报问题后，应优先基于日志和复现情况开展分析。

## 6.2 标准处理路径

1. 有 log：组织对应 Owner 直接分析
2. log 不全：先安排复现
3. 能复现：补齐证据后继续分析
4. 不能复现：联系 TQC 协助复现并抓 log
5. 必要时：安排远程联调

## 6.3 处理口诀

有 log 先分析，无 log 先复现；复现不了找 TQC，复杂问题远程联调。

## 6.4 管理要点

- 不要让“无 log 问题”长期空转
- 不要在无法复现的情况下反复猜测原因
- 复杂问题应尽快进入联合分析模式

# 7. 问题评审机制

不同区域的评审路径不同，应提前明确沟通对象和升级路径。

## 7.1 区域评审方式

| 区域 | 评审方式 |
| --- | --- |
| EU | 通过接口人沟通，必要时拉相关人员协助 |
| AR | 通过接口人沟通 |
| BR | 邮件或微信群沟通 |
| NA | 由 SPM 组织质量 SE、TQC、产品 SE 联合评审 |
| JP | 每周四与 TQC 固定会议沟通 |

## 7.2 管理要点

- 提前明确区域接口人
- 提前明确问题升级路径
- 提前明确固定会议或评审节奏
- 避免多方并行沟通导致结论不一致

# 8. 风险控制重点

## 8.1 时间风险

若未提前规划 TQC 周期，可能导致：

- 版本延迟释放
- 区域项目节点延期
- 关键市场交付受阻

## 8.2 质量风险

若带着 block 问题进入 TQC 或最终释放，可能导致：

- 场测失败
- 区域拒绝接收
- 质量风险升级

## 8.3 协同风险

若日志、复现、评审路径不清晰，可能导致：

- 问题闭环缓慢
- 区域与总部沟通低效
- 责任边界不清晰

# 9. 标准管理要求

## 9.1 三个必须

- 必须在软件释放前完成 TQC 提测
- 必须确保 TQC 阶段无 block 问题
- 必须提前规划测试计划和沟通路径

## 9.2 三个避免

- 避免未沟通测试计划就临时提测
- 避免问题无 log、无复现直接空转
