# **突发紧急问题攻关SOP**

**一、突发问题范围：**

1、工厂发现问题

2、售后一级响应问题

3、大V评测反馈问题 

4、认证实验室反馈问题

5、Dogfood或TQC反馈问题

**二、对策
**

1、明确问题影响范围

（1）如果是生产问题，拉通DFM团队（主导）、制造代表、AOE确认问题是否必现、是否有日志、产线操作是否有规避措施、是否会导致停线、是否会影响机器正常出货（如果影响正常出货，需要拉通PMO对齐大货贴片时间、走货计划，以便制定补救措施）。

（2）如果是售后或大V问题，与售后服务确认发生问题时间、版本信息、问题数量、问题现象（图片或视频）及概率、问题日志、以便评估当前版本是否暂停升级或生产。

（3）如果认证问题，与认证经理确认具体问题用例、是否影响认证结果、是否需要重新提供认证版本，与认证测试确认用例自测结果。

（4）如果是Dogfood或TQC问题，与google对接人或TQC接口人确认问题发生现象和场景（图片、视频）、问题日志和概率，确认问题是否影响版本正常释放。

备注：如果突发问题直接影响软件释放、产品正常出货、售后一级响应，项目组需最高优先级收集问题信息、推动问题闭环以及同步上升对应部门长。

2、锁定资源制定行动项 

（1）明确问题责任田：如果问题模块明确，可以根据模块责任田确认owner，如果问题模块不明确，先从问题发生场景，log相关报错信息，操作涉及的模块出发，大体确认问题相关onwer（宁可错抓，不可漏网一人）。

（2）快速建立沟通群（包含模块owner、模块se、问题接口人、SQA、项目组成员），拉会对齐行动项，及时更新行动项进展（信息及时群里同步，一天至少需要组织2次会议对齐，确保问题事项按计划进行） 。

（3）问题owner根据现有信息（问题现象或log）输出手动模拟或压测建议、抓取日志指令等。

（4）SQA和问题提报者对齐复现手法，并结合问题owner输出的测试建议和抓日志的指令进行复测，同时对比竞品或历史版本。（工厂问题和AOE对齐问题手法；售后或大V问题和服务代表或客户对齐手法；认证问题和认证实验室对齐手法；TQC或dogFood问题和当地TQC对齐手法）

（5）如果已经分析到问题原因，问题owner需要输出问题根因、解决方案、影响范围、风险点以及详细解决计划（比如几点能自测通过输出临时版本给sqa或售后验证，几点提交代码），并提供临时版本给前端验证（工厂问题版本给AOE验证；售后或大V问题版本给售后服务代表安排验证；认证问题版本给认证组或实验室验证；TQC或dogFood问题版本给TQC验证）

（6）如果问题还未复现或未找到原因，问题owner、模块se、产品se以及其他相关方需要讨论输出、debug实验方向、以及代码检查，寻找突破口（如有必要夹版本缩小可疑修改点），同时要求模块owner到现场分析（工厂问题需到产线现场分析； 售后、大V问、TQC、dogFood、认证问题先预约对方时间进行远程联调，如果远程无法解决owner需出差现场分析）

（7）如果问题解决周期无法满足客户要求，及时上升到对应的产品线部门长以及问题模块部门长，同时做好风险应对措施，避免事态继续扩大 比如：

        a、工厂问题：内部准备新版本，如果版本无法及时释放工厂或产线问题没有规避措施涉及产线放空或返工，项目组汇总问题信息（包括解决计划）上升到产品线部门长，优先找产品线沟通是否可以推迟生产或让步生产，问题通过OTA升级迭代避免影响正常出货。针对工厂不接受让步的问题当问题解决后SPM需要拉通资源，当天出版本，当天测试完成，并且当天释放。

       b、售后问题：先暂停线上版本升级部署，内部准备新版本，然后项目组、问题owner和售后服务代表对齐信息，由服务代表统一对外输出，安抚客服情绪。（比如告知可以手动先关闭某个功能、我们的升级计划等）。

       c、认证问题：内部准备新版本，认证经理优先和google或netflx等厂商谈豁免，如果无法豁免和实验室对齐测试资源，确认新版本能否做最小化测试（迭代测试或问题回归），快速切换版本

       d、TQC或DF问题：内部准备新版本，输出解决计划给TQC先谈让步释放，如果无法让步释锁定TQC测试资源快速切换新版本测试

       备注：如果资源瓶颈导致行动项无法及时闭环需及时上升对应部门长，如果问题无有效解决措施需要上升seg进行攻关

 （8）通用性问题需要同步到其他项目，并由模块owner进行举旗。产品SE内部进行信息同步。

**三、复盘总结**

每一次紧急攻关都是对质量的一次挑战，一场惊心动魄的战役后需要打扫战场，总结成败。记录问题如何发生的，如何改进的，避免重复踩坑。
