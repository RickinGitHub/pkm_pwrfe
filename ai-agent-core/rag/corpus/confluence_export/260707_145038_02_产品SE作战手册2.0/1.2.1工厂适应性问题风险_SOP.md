# 1.2.1工厂适应性问题风险 SOP

> pageId: 583197315 | 导出时间: 2026-07-07T14:51:17.411496

# **SOP简介：**

**文档主要内容：**用于介绍工厂生产适应性问题的识别、评估、处理与闭环方式，统一各角色对“是否影响生产”的判断标准，保障软件版本在导入工厂前具备可生产性，并降低停线、返工、版本混乱及质量风险。

**文档适用角色：**AOE、SPM、研发工程师、模块owner、DFM owner、PQA、开发代表

**适用项目阶段：**Design、LR、TR

## 核心红线

- 工厂软件导入前，必须完成 DFM 测试
- 试产阶段允许存在已知问题，但必须明确备注风险
- 量产预抄写阶段，DFM 测试必须通过
- 不允许带着影响生产的问题进入量产
- 所有生产迭代版本必须经过测试释放后方可导入工厂

---

## 一句话理解

试产可以带着“可控问题”推进，量产不能带着“影响生产的问题”上线。

---

## 一、背景说明

在软件提供给工厂前，需先完成一轮 DFM 测试。
试产阶段一般允许存在部分已知问题，尤其是测试平台不完备的项目，但在给工厂预抄写时必须明确备注已知问题及相关测试风险。

对于量产预抄写版本，DFM 测试必须通过，不能存在影响生产的问题。
若 DFM 测试中发现问题，需拉通 DFM Owner、AOE 进行评审，确认是否影响生产：

- **评审结论为不影响生产**：允许提供预抄写版本给工厂
- **评审结论为影响生产**：必须先解决问题，再重新出版本

---

## 二、角色职责

| 角色 | 职责说明 |
| --- | --- |
| AOE | 组织工厂问题评审，协调资源，推进问题闭环 |
| SPM | 统筹项目节奏与风险协调 |
| 研发工程师 | 负责问题分析、修复和版本支持 |
| 模块 Owner | 负责所属模块问题定位、分析和整改 |
| DFM Owner | 负责 DFM 测试结果判定，评估问题是否影响生产 |
| PQA | 参与质量风险评估和放行评审 |
| 开发代表 | 负责开发侧协调与响应推进 |

## 三、预抄写 PID 策略

每个机芯可能对应多个项目，尤其外销项目更复杂，因此需要根据以下维度判断是否可以共用预抄写版本。

| 维度 | 说明 | 管控方式 |
| --- | --- | --- |
| 区域 | 不同区域由于 DTV 制式不同，需要区分单独预抄写，如 EU（AP/RU/AU/CO/ME/TW/IN）、CA、LA、HK、JP、NA | SID 管控 |
| eMMC 大小 | 对于不支持预抄写 data 分区自适应的机芯，不同 eMMC 大小需要单独预抄写 | 编译管控 |
| 内置视频 | 内置视频在预抄写时打包，不同内置视频需要单独预抄写 | 编译管控 |
| Local Dimming | LDM 与非 LDM 机型需要单独预抄写 | HID 管控 |
| 光感配置 | 是否支持光感、不同光感类型、不同光感配置（IIC）需要单独预抄写 | HID 管控 |
| 功放配置 | 需确认功放配置是否可复用，包括类型、个数、IIC | HID 管控 |

## 四、工厂生产常见问题及排查策略

### 排查原则

先环境，再硬件，再配置，后软件。

| 常见问题 | 排查策略 |
| --- | --- |
| Gamma 调试异常 | 一般为画质参数修改影响，联系对应画质工程师分析确认 |
| 遥控器检测异常 | TCL 产品存在 RCA、NEC、AEHA 协议，不同区域可能使用不同协议，请 AOE 确认遥控器协议是否正确 |
| 预制频道异常 | 确认使用的 SID 是否正确 |
| 远场语音测试异常 | 1）确认硬件是否正常；2）确认远场语音开关是否打开；3）确认 HID 配置是否正确，如 MIC 类型、MIC 个数；4）查看软件录音是否正常 |
| 切 PID 失败 | 排查软件版本是否支持对应 PID，试产软件中可能出现该问题 |
| WIFI 检测不过 | 1）确认硬件是否正常；2）确认工厂环境是否正常；3）排查软件问题 |
| Local Dimming 测试异常 | 一般为 LD 配置问题，联系对应 Owner 排查配置 |
| 亮度不达标 | 一般为画质参数配置异常 |
| 灵敏度不达标 | 一般为硬件参数配置异常 |
| 能效不达标 | 一般为硬件参数配置异常 |
| PDIM、ADIM 电压异常 | 一般为硬件参数配置异常 |
| 信源、信号异常 | 1）先排查工厂环境；2）联系 Owner 分析 |
| 声道测试异常，如天空音、重低音无声 | 1）排查电声配置是否正常；2）确认测试硬件及测试手法是否正确；3）由软件 Owner 分析日志 |

## 五、生产过程工厂提报问题处理机制

### 5.1 仅 P 模式存在问题

对于仅在 P 模式下存在的问题，可拉通以下角色共同评估并制定临时方案：

- AOE
- DFM 开发 Owner
- 模块 Owner

处理思路：

- 提供临时解决方案
- 通过调整生产流程、测试方法等方式规避问题
- 在不阻塞生产的前提下推进后续修复

### 5.2 非严重影响用户使用的问题

对于不会严重影响用户使用、但在生产中需要关注的问题，需拉通以下角色进行评审：

- AOE
- PQA
- 开发代表

处理要求：

- 对问题进行联合评审
- 制定改善计划
- 制定售后应对策略
- 达成“不阻塞生产释放”的结论后方可放行

### 5.3 涉及返工的严重问题

对于影响生产并可能导致返工的严重问题，处理原则如下：

1. 第一时间上升至部门长
2. 成立问题攻关小组
3. 问题解决后输出紧急版本
4. 协调测试团队紧急测试和释放
5. 组织返工及生产版本迭代

---

## 六、工厂低概率问题处理原则

生产过程中可能出现低概率问题，这类问题可能需要较长周期才能彻底解决。
对此类问题，应拉通以下角色评估：

- AOE
- 制造代表
- 开发 Owner

处理原则：

- 先评估是否阻塞生产
- 若评估结论为不阻塞生产，可先支持生产继续进行
- 同步推动开发 Owner 开展问题分析
- 问题解决后纳入后续生产迭代版本

---

## 七、各阶段样机生产适应性问题处理方式

### 7.1 试产阶段研发样机

在生产过程中遇到问题时，处理方式如下：

1. 工厂邮件通知产品 SE
2. 软件相关问题由产品 SE 回复工厂，可先跳过相关流程
3. 研发相关问题由硬件侧回复处理结论
4. AOE 提交问题到项目 JIRA 空间持续跟进

### 7.2 外送样机

#### 场景一：问题可通过升级解决

- 可邮件回复工厂先跳过相关流程
- 由 AOE 提交问题到项目 JIRA 空间跟进
- 后续通过升级版本解决

#### 场景二：问题不能通过升级解决

例如 MID key 抄写异常等关键问题：

- 必须先解决问题
- 提供新版本进行返工升级

---

## 八、预抄写软件生产迭代要求

随着需求导入和缺陷修复，生产软件可能需要持续迭代。
所有用于生产的迭代软件必须满足以下要求：

- 所有迭代软件必须经过测试释放（SQA、PQA、TQC、Google(GTV)、Amazon(FTV)）
- 必须包含 DFM 测试通过结论

---

## 九、管理要求总结

### 三个必须

- 必须先完成 DFM 测试
- 必须评审问题是否影响生产
- 必须对生产软件迭代进行受控管理

### 三个禁止

- 禁止带着影响生产的问题进入量产
- 禁止未经测试释放的软件直接导入工厂
- 禁止问题未形成结论就直接放行生产
