# 1.5.3 SCCB评审SOP

> pageId: 597565974 | 导出时间: 2026-07-07T14:53:25.185983

# **SOP简介：**

**文档主要内容：缺陷/风险无法在当前节点按常规路径闭环（修复并验证），通过 SCCB 进行跨角色评审，明确“可接受的闭环方式、风险承担与后续行动”，避免问题被动遗留或争议拉扯。**

**文档适用角色：**大产品SE（机芯软件负责人）& 小产品SE（派生软件负责人）

适用项目阶段：SR3，SR4，SR5，SR6

环境依赖：具备 Jira 访问权限；项目已建立统一缺陷看板/过滤器（建议按版本、阶段、标签聚合）

# SCCB评审 SOP

# 1.什么是SCCB

##    1.1 SCCB定义    

    SCCB（Software Change Control Board）：用于对“无法按常规修复闭环/存在争议/需要风险接受或变更控制”的问题进行评审决策的机制。它不是“关闭问题的手段”，而是对风险与变更作出组织级决策的流程。

sccb成员一般由质量SE，产品SE，VPM，问题owner组成。

##   1.2 评审要解决的三个核心问题

1. 要不要修（必须修 / 可延期 / 不修但需监控）
2. 怎么闭环（修复导入 / 风险接受 release with risk / monitor / 转需求 / 三方提单等）

##   1.3 SCCB 的典型输出（必须落到 Jira/纪要）

- 结论类型：SCCB close / Monitor / 转More info/ assgin
- 结论说明：重点说明采取最终结论的原因

# 2.哪些类型问题需要走SCCB

        首先sccb状态不是有开发owner来改变，是需要由开发owner组长根据开发的分析说明意见来评估是否可以走sccb，只有组长才有sccb权限。

## **2.1 More Info / 未复现类（证据不足导致无法修复）**

**触发条件**（满足其一即可提 SCCB）：

- 开发基于现有信息无法定位（日志缺失/复现步骤不完整/环境不清晰）
- 测试侧按约定的复现方法在**至少两个版本/两轮**尝试后仍无法复现/无法补齐证据
- 问题严重度高但概率低，需要决定“是否压测/是否监控放行/是否阻断过点”

**SCCB关注点**：证据链是否闭合、复现投入是否足够、是否转为监控放行的条件。

## **2.2 架构/方案限制类（短期无可行修复或修复风险过高）**

**触发条件**：

- 根因明确但修复需要大改架构/跨模块联动，当前版本窗口无法承受
- 现阶段无可行方案或方案风险显著高于问题本身（回归/稳定性/性能/认证风险）
- 历史版本长期存在，需要组织确认“接受遗留/规划版本修复/规避方案”

**SCCB关注点**：是否真实为架构限制、是否有替代方案、是否可以通过配置/规避/降级降低风险、是否需要形成长期整改计划。

## **2.3 争议类（开发认为无效/非缺陷，测试不认可）**

**触发条件**：

- 同一问题在“是否缺陷/严重度/归属/是否可接受”上分歧明显
- 需要产品/质量/业务方对用户影响与需求边界做裁决
- 可能涉及：需求理解差异、用例不合理、设备异常、环境差异、三方限制等

**SCCB关注点**：基于事实证据做裁决（需求边界、验收标准、复现证据），并明确后续动作（改用例/转需求/设备隔离/补证据等）。

# **3. SCCB发起与权限（建议写成流程动作）**

- **发起建议**：开发Owner/组长、质量SE/SQA、产品SE均可“建议进入SCCB队列”。
- **状态变更/结论落地**：由 SCCB 会议达成一致后执行（避免单方用 SCCB 作为“关单手段”）。
- **必备输入**：未满足输入项不得评审通过（见下文“输入清单”）。

# **4. SCCB评审输入清单（建议作为 Jira 评论模板）**

每个进入 SCCB 的问题，至少补齐：

1. 影响描述：用户场景/认证/量产/售后影响，是否 Block
2. 复现信息：平台/PID/版本、复现步骤、复现概率、对比机
3. 证据：log/截图/视频/抓栈/统计数据（缺什么写清楚）
4. 分析结论：初步根因/怀疑点/排查范围
5. 方案与风险：修复方案、规避方案、修复风险、回归范围
6. 建议结论：建议 Fix / Monitor / Defer / 转需求 等 + 理由

# **5.针对三类 SCCB 问题：产品SE需要做什么（可执行动作）**

## **    5.1 More Info / 未复现类：产品SE动作清单**

      目标：避免“没证据就放行”或“无效投入过度压测”。

**产品SE需完成：**

1. **复现一致性校验**：确认复现人员/方法是否一致（是否同步过复现手法、环境、步骤、资源文件）
2. **环境一致性校验**：平台/PID/版本/配置是否一致；必要时要求补“可复现实验矩阵”（版本×设备×场景）
3. **严重度驱动策略**：

- 高严重度/高风险：推动制定“模拟场景复现方案”（与开发一起定义触发条件、插桩、抓取点）
4. 低概率但严重：推动按质量策略压测（如 ≥3台、24h 或项目约定）并明确通过标准
5. **闭环路径建议**：若仍无法复现，提出是否允许 Monitor 放行及其前提：
监控指标（Crash/重启/ANR/关键日志）
6. 触发阈值与回滚策略
7. 下版本必须补齐的证据/行动项

## **5.2 架构/方案限制类：产品SE动作清单**

目标：防止“用架构限制当挡箭牌”，同时把“遗留风险”变成可追踪计划。

**产品SE需完成：**

1. **要求专业背书**：必须拉通相关领域专家/SEG形成联合会诊结论（纪要落 Jira）
2. **历史对比验证**：确认是否历史版本一直存在，或是否由某需求/版本引入（必要时要求回归到“未导入需求”基线验证）
3. **替代方案/规避方案评估**：是否可通过配置、降级、提示、灰度、限流等降低风险
4. **风险接受条件**（若决定不修/延期）：明确
影响范围与可接受理由
5. 监控方式与指标
6. 修复规划版本/里程碑（不能只写“后续看情况”）

## **5.3 争议类：产品SE动作清单**

目标：把争议变成“证据 + 标准”的裁决，避免长期拉扯。

**产品SE需完成：**

1. **先判定边界**：这是缺陷、需求、用例问题还是设备/环境问题（必要时拉 BA/业务方）
2. **证据裁决**：要求双方提交可核验材料（复现视频/日志/对比机/标准条款），并在评审中形成结论
3. **分类落地动作**：
新需求/体验取舍：推动 **转需求**，写清验收标准与版本计划
4. 设备异常：推动设备登记隔离（避免重复投入）
5. 用例问题：推动 SQA 修订用例/补充前置条件
6. 真实缺陷：明确 Owner 与修复计划，必要时提升优先级/阻断过点

注：争议类通常不建议“直接 SCCB 关单”，更常见的输出是“补证据/转需求/修复计划”。

# 6. 结论强调（建议保留但更可执行）

SCCB 不是为了关单，而是为了**在无法常规闭环时做组织级决策**。
- 任何 SCCB 结论必须同时包含：**结论类型 + 风险说明 + 行动项 + 验证/监控方式**，否则视为未闭环。
