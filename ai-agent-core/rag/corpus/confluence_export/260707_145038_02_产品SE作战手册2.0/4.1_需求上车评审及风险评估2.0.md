# 4.1 需求上车评审及风险评估2.0

> pageId: 583197702 | 导出时间: 2026-07-07T14:50:54.664333

| 标题 | **需求上车评审及风险评估** |
| --- | --- |
| 状态 |  |
| 角色 | A |
| 更新时间 |   |
| 沟通对象 | 领域SE、需求分析师（BA）、软件产品SE、软件项目经理、开发工程师、测试工程师、质量SE |
| 具体内容 | 主要从合规（三分支原则，领域SE+2， 需求需要上车平台通过报告），影响可控  1. 参与新需求上车评审  2. 评估需求是否存在风险  3. 评估是否存在Block/重点问题，对整机软件影响  4. 需求代码导入时间确认  5. 需求上车SE需关注的信息  6. feature隔离影响 |
| 操作方法 |      1、检查需求评审上车要素表是否齐全（需求矩阵、测试用例、自测报告、SQA测试报告、性能、稳定性、安全、集成说明）  2、确认需求矩阵设计是否符合用户操作、使用习惯  3、确认测试用例是否覆盖完全  4、检查自测、SQA测试报告是否通过，对未关闭问题进行评审，确认是否有未解决问题，如有不允许上车；  5、性能、稳定性如有劣化不允许上车  6、涉及安全问题的需整改后才允许上车  7、根据集成说明进行Feature功能配置，整机系统相关配置，安排CIE集成导入；     1.需要确认需求代码导入时间   - 代码导入（Merge）时间的随意性会导致集成冲突、版本构建失败或测试验证延期，需要明确时间节点以便拉齐各方节奏。 - **应对策略**：  - **制定时间窗口**：设定明确的代码提交与合并时间窗口（如每日下午4点前完成评审，5点统一触发集成构建）。 - **建立倒推机制**：根据项目发版节点（Release Date），倒推测试排期，从而严格锁定代码冻结（Code Freeze）和导入的最终Deadline。 - **风险预警**：对于接近导入时间节点仍未完成评审的评审项，通过自动化工具或项目经理进行红灯预警。   2.需要明确需求上车，SE需要关注哪些信息   - SE（系统工程师）在需求上车把关时，如果缺乏标准化关注列表，容易遗漏关键风险点（如性能退化、内存泄漏、安全合规等）。 - **应对策略**：  - 明确SE必查维度，建议包含：  1. **合规性**：三分支原则是否遵守？依赖的底层接口是否变更？ 2. **质量门禁**：自测报告、SQA测试报告是否100% Pass且无遗留Block/Major问题？ 3. **非功能性指标**：性能、稳定性、安全评估是否达标（无劣化）？ 4. **配置说明**：Feature的集成配置说明是否清晰准确？   3.灵悉和非灵悉项目，feature隔离影响   - 不同项目基线（灵悉 vs 非灵悉）对同一Feature的诉求不同，若代码未做好隔离，极易发生“A项目上车导致B项目崩溃或逻辑异常”的交叉污染。 - **应对策略**：  - **强化宏隔离/Feature Toggle机制**：要求开发人员必须使用系统级Feature宏开关或配置文件对新功能进行严格隔离。 - **双向回归验证**：测试环节必须包含“开启该Feature（灵悉）”和“关闭该Feature（非灵悉）”的双态回归测试，确保非目标项目不受任何影响。 - **代码Review重点审查**：模块SE在代码评审时，将“是否有越权修改通用逻辑”、“开关逻辑是否严密”作为一票否决项。   4.二资源切换也属于需求，在code上车前也需要明确互锁。   - 二供（第二供应商）资源切换往往被研发人员误认为是单纯的“底层替换”而轻视，实际上它牵涉极大的整机稳定性和兼容性风险，必须按完整需求流程管控。 - **应对策略**：   - **强制交付件审核**：二资源代码上车前，必须补齐《需求上车要素表》中的所有内容，特别是性能、稳定性自测报告和SQA交叉验证报告，SE在Review时按新需求标准执行“强互锁”拦截。 |
| 标准与原则 | 新需求功能符合用户体验，交付件齐全，方案测试通过，性能、稳定性、安全符合设计要求。 |
