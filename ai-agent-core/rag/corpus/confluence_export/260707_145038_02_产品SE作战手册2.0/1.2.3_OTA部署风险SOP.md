# 1.2.3 OTA部署风险SOP

> pageId: 583200928 | 导出时间: 2026-07-07T14:51:17.998924

# **SOP简介：**

**文档主要内容：**我们软件释放后需要进行ota部署，部署前需要对所部署的机型进行ota测试，测试通过后才能部署。

**文档适用角色：**SPM，产品SE，质量SE，VPM，在线质量Owner

**适用项目阶段：SR6**

**环境依赖：**

**相关内容链接：**

**[升级放量规范V4](https://confluence.tclking.com/pages/viewpage.action?pageId=489724214)**

[在线质量平台](https://taijibi.tcl.com/webroot/decision#/?activeTab=dec-tabs-homepage)

## 一句话理解

OTA 部署不是版本发布后立即上线，而是必须经过测试、审批、部署、放量和上线监控的完整风险管理流程。

## 核心红线

- 未完成 OA 释放单签署，不允许发起网络升级测试申请
- 未完成 OTA 测试验证，不允许进入部署
- 部署 PID、历史版本信息必须与 SQA 测试报告一致
- 未经过放量评审通过，不允许正式放量
- 若存在严重售后问题或在线质量指标劣化，必须上升决策是否暂停部署

# 1. 背景说明

软件释放后，产品通常还需要进入 OTA 部署流程。
OTA 部署并不等同于直接面向用户发布，在正式部署前，必须先对计划部署的机型进行 OTA 测试，验证升级链路、目标版本及历史版本路径的可用性。

OTA 测试通过后，版本方可进入部署流程。
完成部署后，还需根据放量规范组织评审，并持续关注线上激活、售后和在线质量数据，以降低升级风险和用户影响范围。

# 2. 整体流程总览

OTA 风险管理整体流程如下：

1. OA 释放单签署完成
2. 产品SE发起 OTA 升级测试申请
3. SQA 完成网络升级测试
4. 测试问题评审
5. 发起 OA 部署流程
6. SPM 跟进 OA 签批
7. SQA 执行部署
8. 组织放量评审
9. 正式放量
10. 上线后持续监控

# 3. 角色职责

| 角色 | 职责说明 |
| --- | --- |
| SPM | 组织测试问题评审、推进 OA 流程、组织放量评审会议、跟进整体部署节奏 |
| 产品SE | 发起 OTA 升级测试申请、发起 OA 部署流程 |
| 质量SE | 参与放量评审、关注质量风险并输出结论 |
| VPM | 参与放量评审与版本风险判断 |
| 在线质量Owner | 监控上线后在线质量数据，参与风险评审 |
| SQA | 执行网络升级测试，输出测试报告，并在审批完成后执行部署 |

# 4. OTA链路测试管理

## 4.1 发起前提

仅在 OA 释放单签署完成后，方可发起网络升级测试申请。

OA 释放单签署完成包括以下条件：

- 内部释放完成
- TQC 释放完成
- Google approval 完成(GTV)
- Amazon 完成释放(FTV)

## 4.2 测试申请

由产品SE发起 OTA 升级测试申请，申请内容应包含：

- 所有需要部署的 PID
- 历史版本信息

## 4.3 测试问题处理

若 SQA 在测试过程中发现问题：

- 由 SPM 组织相关方进行评审
- 评审通过后，方可继续推进 OA 释放单及后续流程

# 5. OTA部署计划管理

## 5.1 部署前提

以下验证或释放完成后，产品SE方可根据 SQA 网络升级测试报告发起 OA 部署流程：

- SQA 释放完成
- TQC 释放完成

- EU
- JP
- AR
- BR
- NA

- GTV NA Google dogfood 完成（根据 Google 要求）
- FTV 项目 Amazon 要求完成（如适用）

## 5.2 关键信息一致性要求

部署时必须确保以下信息与 SQA 测试报告一致：

- 部署 PID
- 历史版本信息

## 5.3 OA签批推进

- SPM 负责跟进 OA 流程签批进展
- OA 签批完成后，由负责部署的 SQA 执行部署

## 5.4 特殊区域说明

- 印度区域因供电不稳，一般进行单独部署管控

# 6. OTA放量计划管理

## 6.1 放量依据

OTA 放量应参考：

- [升级放量规范V4](https://confluence.tclking.com/pages/viewpage.action?pageId=489724214)

## 6.2 放量评审机制

由 SPM 组织放量评审会议，参会角色包括：

- 质量SE
- 产品SE
- VPM
- 在线质量Owner

评审通过后，方可执行放量。

## 6.3 特殊放量场景

若评审未通过，但因特殊情况仍需放量：

- 必须上升产品软件部部门长审批

# 7. OTA部署后监控机制

OTA 部署完成后，仍需持续监控上线表现。

## 7.1 激活情况监控

重点关注：

- 新版本激活情况
- 升级成功情况
- 激活趋势是否异常

## 7.2 售后数据监控

重点关注：

- 是否出现共性严重问题
- 是否存在聚集性售后异常
- 是否出现升级后明显投诉增加

## 7.3 在线质量监控

通过在线质量平台重点关注：

- CPU 高负载异常
- 低内存比例异常
- 其他关键维度是否劣化

## 7.4 严重风险升级机制

若部署后反馈严重售后问题或在线质量维度明显劣化，包括但不限于：

- 升级卡 logo
- 聚集性售后问题
- 一级响应问题
- CPU 高负载异常
- 低内存比例异常

则需及时上升软件产品部部门长，由其决策是否暂停部署。

# 8. 风险控制重点

## 8.1 测试风险

若未完成 OTA 测试即直接部署，可能导致：

- 升级失败
- 历史版本无法正常升级
- 部署路径与测试路径不一致

## 8.2 配置风险

若部署 PID 或历史版本与测试报告不一致，可能导致：

- 测试结论失效
- 实际部署风险无法评估
- 问题扩大到真实用户

## 8.3 放量风险

若未经过放量评审即直接放量，可能导致：

- 风险问题快速扩散
- 售后压力激增
- 在线质量指标大幅劣化

## 8.4 上线后监控风险

若部署后缺乏持续监控，可能导致：

- 严重问题发现滞后
- 错过最佳止损窗口
- 用户影响范围扩大

# 9. 标准管理要求

## 9.1 三个必须

- 必须在 OA 释放单签署完成后才能发起 OTA 测试申请
- 必须保证部署信息与 SQA 测试报告一致
- 必须经过放量评审通过后才能正式放量

## 9.2 三个重点关注

- 关注历史版本升级链路是否完整
- 关注部署 PID 与测试对象是否一致
- 关注部署后售后与在线质量数据变化

## 9.3 三个避免

- 避免测试未完成就提前部署
- 避免测试报告与实际部署配置不一致
- 避免部署后不监控激活、售后与在线质量数据
