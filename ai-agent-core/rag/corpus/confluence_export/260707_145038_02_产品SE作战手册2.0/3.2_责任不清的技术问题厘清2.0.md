# 3.2 责任不清的技术问题厘清2.0

| **标题** | 责任不清的技术问题理清2.0 |
| --- | --- |
| **状态** |  |
| **角色** | A |
| **更新时间** |   |
| **沟通对象** | 各软件模块Owner、模块SE、SEG、产品SE、SPM、SQA |
| **具体内容** | 问题分析->责任界定->问题攻关、收敛 |
| **操作方法** | **一、问题分析**  1）产品SE需要对齐问题复现手法和环境，并安排sqa复现问题现象，现场debug抓取log等。    ①产品SE协助理清影响问题的环境因素。   如确认pid，网络环境，软件版本，操作手法等。如果为异常操作需要和测试开发及时澄清。对jira上issue单sqa备注的信息进行对齐，一定要重新确认概率，恢复方法，使用场景等，排除干扰开发owner判断项。    判断是否是单台机器问题，排除硬件异常。使用多台机器进行对比测试，或者将异常机器的软件dump出来，反抄到正常机器中，看是否依然有问题。    ③如果遇到需要复现的问题，协调项目组拉通测试资源安排压测，记录压测数据，判断问题发生概率。定期收集好压测数据，对各个平台进行沉淀，确认是否有同样场景，为后续类似问题判别提供快速有效的log信息等。     2）产品SE和开发owner初步分析log找出问题切入点。     如何确认切入点开发owner：     • 例如遇到了搜台无节目，那么可以先从上层应用owner开始看起，然后就是中间件接口，和底层soc。从上往下分析。    • 例如画质相关问题，颜色有异常闪屏等情况，可以让soc先分析，用工具打pattern等方式，从下往上分析。    • 在哪一环出现了问题，就由这一环节的owner作为切入点owner。    3）根据切入点owner分析和责任田文档找出关联模块owner，一起查看现场环境或者log给出各自的初步分析，排查方向和行动项。    ①切入点owner和se需要经过问题初步探讨，确认相关联模块，并拉通关联模块owner介入分析。    ②各开发owner应该需要针对问题，给出下一步行动项，例如对比实验，加log debug，检查代码，查看需求文档等。      **二、责任界定**  1）各关联模块owner给出初步分析和行动项后，SE和PM联系关联模块负责人，拉会讨论。     ① 引入快速响应机制，建立攻坚小组，点对点、面对面和模块owner推进。    ② 明确“瓶颈环节”，确认问题处理周期。  2）产品SE主导会议，和各模块owner讨论问题的发生原因，执行逻辑，参数配置等。提供一些解题思路和建议，区分是软件责任还是硬件责任。    ① 一般问题责任可能是软件问题或者硬件问题，需要在这一环理清软硬件责任。    ② 区分软硬件责任，可以做一些对比交叉实验，且最好要保留问题板卡的环境不被破坏，举例51A emmc案例：    RT51A6月份遇到了一个售后问题，emmc电压异常，导致机器会经常性卡死。因为复现概率较高，且前端已经确认了复现手法，开发owner从软件排查，没有发现明显异常信息。便进行试验，将问题机器软件dump出来，反抄到其他的批次机器中，按照同样的手法进行压测复现，通过压测数据证明其他批次无异常，需要硬件接力分析。最后经过硬件分析，对比两批次差异点，确认出了emmc电压部分不同，并找出了问题根因。    ③ 如果遇到无法界定软硬件责任，且概率较低问题，需要协调资源继续安排压测，保留问题板卡，软硬件联调分析，并给出行动项排期。   需要确认交叉验证实验需要由硬件制定，提前指定好需要做的交叉实验，减少临时设计实验和未覆盖到的场景。     3）如确认为软件，且各模块间有异议，模块owner需要提供log，代码片段，需求文档，实验数据等相关证据，SE协助进行理清。    ① log 和 代码片段：安排线上会议一起看代码逻辑，并根据抓取的log，展示程序执行流程的code，大家一起判断这一块的设计是否合理，或者有异常的地方可以及时指出。    ② 需求文档：如果某一环节，有需求文档，就是要按照这样的方式设计或者逻辑是需要这么处理，那么需要让其他的模块判断是否要兼容修改。    ③ 实验数据：各模块owner可以根据自己行动项，或者自己增加的调试log，将实验数据给出，证明自己的模块是否异常。    ④ 参考实例：   RT51A 去年7月份，遇到一个量产搜台问题，soc在搜到台后，多次下解遮黑命令，但是中间件接到msg后，对消息的处理异常，导致搜台有问题。 soc  owner认为此处只是消息的消费者，给多少次命令，就需要处理多少次，他们多次下命令，也是需要中间件处理好。但中间件owner认为，命令发送不应该是根据lock channel 回调多次给命令。两方都认为对方的处理方式不妥，需要进行整改，此时se就给出计划，让开发在逻辑流程中加log，并修改部分逻辑，提供软件协调sqa资源进行测试和数据收集。最后根据结果来看，是需要soc对于命令发送进行处理。但此时soc依然有异议，最后就上升评审，商讨最终解决方案由SOC加回调逻辑。     4）如无有效结论，产品SE根据实情主导及时上升到模块SE或SEG（[SEG攻关流程](https://confluence.tclking.com/pages/viewpage.action?pageId=234111361)），邀请加入一起商讨解决方案  5）责任田传递   切入点owner分析后，如果确认问题不在本模块或者需要关联模块同步修改 → 必须提供充分证据（log、代码、实验数据、文档等）， 由产品SE知情，转交给下一棒模块owner。  6）问题梳理整洁    ① 所有分析过程、实验数据、log、代码片段集中记录在，Confluence、Jira上。    ② 问题分析的过程中，每一个阶段分析，形成阶段性小结，为后续问题定位可方便排查。  **三、问题攻关、收敛并跟进闭环**  会议前预收集机制：产品SE会前协助各模块owner备注了解现阶段分析结论、证据、下一步建议，减少会上不必要的思考时间和，问题环境对齐时间。SPM整理会议记录，根据会议讨论的结果，进行邮件输出。  1）产品SE需要提前关注问题当前的进展和下一步计划，以及是否有技术瓶颈导致无法解决，并提前拉通开发owner 资源。  2）责任owner给出具体解决方案，和计划排期，SPM根据排期去跟进责任owner问题闭环。 |
