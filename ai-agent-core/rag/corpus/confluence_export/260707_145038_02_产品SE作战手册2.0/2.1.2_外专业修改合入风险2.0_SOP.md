# 外专业修改合入风险2.0

## 1.背景

在项目开发及版本集成过程中，PQ / AQ / HID / LDM算法等外专业修改会频繁导入。
由于外专业修改通常具有**模块分散、依赖复杂、责任边界不清晰**等特点，若合入评估不足，极易引入版本不稳定风险。

外专业修改的核心风险在于：**修改是否完整合入、依赖是否同步、影响范围是否评估清楚。**

## 2.具体模块

### 2.1PQ/AQ导入

#### 2.1.1 具体事项

1. 产品SE按照PQ/AQ spec进行数据库修改。在进行数据库修改时一定要注意DB路径是否正确，是否与pid绑定。
2. PQ/AQ工程师在[tclconfig配置工具](https://idsaas-o.tclking.com/page/cms-tclconfig/#/home)上修改PQ/AQ参数后及时推送到tclconfig，并提供上code链接，由产品SE入库。

#### 2,1,2 风险

 代码漏合入： PQ/AQ在配置一款新品的时候往往需SOC配合修改代码或者bin，例如tcon bin等，而PQ/AQ工程师往往只负责上传自己的pq bin，没有去care soc部分代码是否有上传。由于这部分是PQ/AQ与soc联调，产品SE不了解整个过程，会导致只merge了部分修改。遗漏的修改会出现点屏异常、声音异常等问题，从而导致版本重发。

#### 2.1.3 措施

1. 在merge PQ/AQ的修改时，一定要PQ/AQ工程师提供jira单，通过jira单我们可以排查到是否有soc修改内容以确保修改的完整性
2. PQ/AQ修改需要产品SE关注是否影响到其他pid，若有需要确认其他pid是否也存在。（一般情况下pq bin会多个项目共用）
3. pq/AQ在做新老品兼容时，一定要pq提供新老屏测试报告以及测试用例场景，全部符合后才能入库

### 2.2HID修改

#### 2.2.1 具体事项

现在的项目都采用了软硬分离的方案，硬件部分（HID）的修改主要是由硬件上code后将修改提供给产品SE，由产品SE入库。

#### 2.2.2 风险

HID配置不对的话会出现屏幕不亮、画异的问题，从而导致版本提测失败，甚至影响版本释放

#### 2.2.3 措施

根据FDS检查HID的以下配置项是否一致，如果不一致，需拉通BA、硬件工程师一起核对，问题澄清后修改才能入库

1.feature 属性是否配置正确

![image2023-11-9_21-2-7.png](attachments/image2023-11-9_21-2-7.png)

2.EDID路径是否配置正确，DLG 、Dolby version 、DTS 等在EDID配置，上层动态识别，通过不同文件隔离。

3.tuner类型是否配置正确

4.功放是否配置正确

5.EXPATH 配置是否正确。根据机型FDS 是否带远场语音以及远场语音灯数量。

6.SOURCE_SUPPORT 机器外设配置是否正确。

7.Tcon型号是否配置正确

8.屏参修改

### 2.3 SOC 闭源KO

#### 2.3.1 事项

soc 部分闭源修改是以ko文件的形势提供的，模块owner看不到其中的内容，只能由产品SEreview后直接合入

#### 2.3.2 风险

 当kernel 涉及到头文件，新增函数修改或者config 打开关闭时，对应ko需要重新依赖新改动的代码进行编译生产，不然会出现ko和kernel不匹配问题 ，从而导致功能失效或异常的问题

#### 2.3.3 措施

识别到修改中包含ko文件时，需与SOC沟通，确认是否有涉及到头文件的修改，如果要，需要求SOC重新依赖新改动的代码进行编译并提交产物，提交后才能一起入库

### 2.4 AIAQ/AIPQ修改导入

#### 2.4.1 事项

AIAQ/AIPQ模块的修改涉及专业组较多，产品SE需要对整体进行把控，包括修改的review、入库，SQA测试，专业组验收等

#### 2.4.2 风险

 当AIAQ/AIPQ模块的修改有问题时，可能会导致声音、画质显示异常，版本提测失败，浪费人力等问题

#### 2.4.3 措施

识别到是AIAQ/AIPQ模块的修改时：

1、先让模块owner对修改进行review，模块owner+2后再入库

2、SQA测试完成后，让画质/电声专业组对软件进行验收
