# **签BOM风险SOP**

## **1、BOM的介绍**

1）BOM是一个项目的物料清单，工厂按照BOM里的物料清单去准备对应物料组装机器。软件没有实体的物料，**软件专业审批BOM就是把软件负责的4个虚拟物料（PID部品/MID部品/主程序/小版本号）打包成一个大的软件部品（D080001），添加进BOM中的步骤**。

2）按项目阶段分类

BOM伴随着整个项目开发过程，在项目的不同阶段期，BOM的状态也不同。

BOM分为：

BOM1：关键件BOM、计价BOM、长周期采购物料备料

BOM3：试产BOM，小批量采购

BOM4：量产BOM，批量采购物料

一个基础的长周期项目才会有BOM的所有状态，小项目可能只有BOM4。软件产品SE不会接触到BOM1，项目到达软件产品SE所接触时，已经处于BOM3或者BOM4的阶段。

3）按BOM的项目类型分类：

- 整机类：一般适用于基础类的项目
- 整机简单派生类：适用于那种base基础类项目，变更部分物料或者功能，派生出新的项目
- 计价BOM：ODFBOM4释放类型BOM产生前，用于报价或者成本核算的 BOM。不需要软件产品SE签批。
- ODFBOM4释放：也叫订单BOM，通过订单形式所发起的BOM，是基于成熟或者已经释放的项目的BOM，直接进入BOM4阶段。

4）BOM周期与项目类型组合：

|  |  | 整机类 | 整机简单派生类 | ODF | 审批流程（仅确认审批） |

| 1 | BOM1 | - | - | - | - |
| 2 | BOM3 | BM11 BOM3制作释放流程（整机类）  BM32 商用BOM3制作释放流程（整机类） | BM10 BOM3制作释放流程（整机简单派生）  BM31 商用BOM3制作释放流程（整机简单派生） |  | BM05 BOM3审批流程（整机类、整机简单派生）  BM12 BOM3转BOM4 |
| 3 | BOM4 | BM18 BOM3制作释放流程（整机类）  BM37 商用BOM3制作释放流程（整机类） | BM19 BOM3制作释放流程（整机简单派生）  BM38 商用BOM3制作释放流程（整机简单派生） | BM44 计价BOM制作释放流程  BM45 ODF-BOM4制作释放流程  BM51 商用ODF-BOM4制作释放流程 | BM14 BOM4审批流程（整机类、整机简单派生） |

*注：BMxx是BOM的流程序号，从1~100，没有具体的意义

为了适应海外工厂的生产需求，有些项目会将一个BOM拆成了2个BOM，一个整机BOM，一个主板BOM，对我们来说按相同的方法签批即可

## **4、ODF BOM的签批注意事项**

ODF BOM，也就是类型为"ODFBOM4释放“的BOM。

（1）这是一种BOM4（即量产BOM）的快速订单BOM，特点就是快，从下订单到出货可能就1~2周，周期非常短，任何软件的delay，都很可能造成项目的延误。

（2）这种BOM在新国家上的派生很常见，因此需要确认知道当前软件针对这个国家是否有给SQA点检过。如果没有，需要在BOM的签批后，立即知会SQA进行国家点检，同时知会整机项目管理、开发代表、软件项目管理当前项目（当前BOM）需要等待软件的点检结果。

（3）正常的ODF BOM不会有新的需求，因此软件组件不需要做任何的变更。但特殊情况下，会存在软件需求变更，此时需要留意项目的需求评估会，明确是否需要进行软件组件及软件释放的变更。

（4）ODF BOM的签批判断与整机BOM有些差异，整机BOM的签批主要是通过各种check list项来选择正确的组件，而ODF BOM的判断主要是通过与参考BOM的差异来进行判断软件主件是否需要修改。主要的判断内容如下：机芯、品牌、产品型号、屏、功放、PID、MID、主软件组件信息的对比、客户机型、国家。

其中，国家又是其中的重点检查项，因为经常会存在参考BOM与ODF 所派生国家对不上，而导致tuner制式对不上，或者没有点检的情况。

## **5、公共抄写属性的添加**

公共抄写属性是所有项目都需要在生产时确认是否需要添加的属性。在完成软件组件添加后，PDM系统强制要求产品SE完成个属性的流程，才允许软件环节往下跑。

由于公共抄写属性与MID组件有些类似，因此公共抄写属性将一部分key移到MID组件里，目前只剩下一项是必填项：机型名抄写

原因是一个PID会对应多个BOM，而BOM里的机型与默认的机型名称不一定相同，因此需要在工厂生产时增加机型名的抄写。

（1）如果BOM的审批流程还没有跑完，可以直接找BOM的审核人：杨琴（yangqin）帮忙直接修改。

（2）如果BOM流程已经跑完，则需要发ECN（工程变更通知）进行变更，具体的操作流程：

（3）ECN变更需要按以下的原则执行

- **跟进ECN流程闭环：ECN/MEMO所有签批，需要在24H内签批完成，原则上按照最高优先级处理**
- **发起、签批、审核环节：需要明确关键信息，且与提供信息方达成一致。若存在模糊信息，需确认后方可审核或签批**
- **信息通知：ECN发起后，立刻知道整机项目管理、软件项目管理及相关人员。对于小于48小时生产的项目，需要微信，邮件，电话，T信等方式通知到位，并确保通知到生产端。**
- **ECN反馈：确保生产机型是否立即生效，并确认TR/PR等过程机器执行到ECN。**
- **ECN执行：确认不适用软件版本是否紧急停用，确认PID、MID在工厂生产流程执行是否顺畅。**
