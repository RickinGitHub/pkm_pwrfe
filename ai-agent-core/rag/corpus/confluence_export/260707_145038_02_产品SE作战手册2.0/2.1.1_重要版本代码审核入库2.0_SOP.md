# **SOP简介：**

## **一、 角色职责矩阵** 

为确保版本合入的严谨性，对各参与角色的职责进行明确界定：

| 角色 | 在版本带入与代码入库中的核心职责 | 权限与考核维度 |
| --- | --- | --- |
| **产品SE ** | 【核心把关人】全盘掌控各机芯需要导入的新需求、配屏、问题修复；负责审核应用的更新范围；主导入库前的最终复核与放行。 | 拥有一票否决权，对版本编译成功率及基础质量负责。 |
| **模块Owner/领域SE** | 负责本模块代码的Review，确保代码规范性；确认合入范围不引起交叉污染和性能劣化。 | 对本模块合入代码的代码质量、安全性负责。 |
| **系统/应用开发工程师** | 严格执行“三分支”入库原则；提供清晰的修改说明、自测报告；涉及路径更换时需出具差异与风险说明邮件。 | 对需求/问题的解决有效性及代码齐套性负责。 |
| **PQ/AQ/硬件研发工程师** | 负责画质/音质/硬件参数的调试与DB修改；**重点**：需主动拉通SOC开发，确保参数Bin文件与底层代码修改同步上传。 | 对音画质/硬件适配参数的准确性及依赖完整性负责。 |
| **CIE（集成工程师）** | 负责接收APK更新并进行系统集成，反馈集成过程中的构建风险。 | 对集成路径的正确性和构建流程稳定性负责。 |

## 当前大部分项目都以灵悉火车运作模式

## **二、 灵悉火车运作模式与带入准则 **

针对不同的发布节奏，设定差异化的带入标准与风险容忍度：

1. **主干版本  —— “全量导入，充分暴露”**

- **定位**：用于全功能测试，发现并解决系统性问题。
2. **带入准则**：除极其特殊的重要节点版本需要冻结外，其他所有开发完成的需求、Bug Fix均需全量导入。
3. **班车版本 —— “聚焦增量，严控风险”**

- **定位**：主要用于导入新需求、配屏适配或季度升级SPL。
4. **带入准则**：配屏参数、新需求（已通过上车评审）以及严重Bug（Block）优先且必须导入；普通问题需评估后按需导入。
5. **Hotfix或量产版本 —— “底线防御，极简变更”**
**定位**：满足项目快速生产与紧急交付。
6. **带入准则**：**实施严格变更控制，**仅允许导入阻塞生产的Block问题和一级售后问题。任何额外的新需求合入必须特批。

## **三、多维度合入内容核查清单**

产品SE在每个版本发布前，需对照以下多维度进行“齐套性与合规性”核查：

**维度1：新需求导入核对**
核对《需求上车评审纪要表》，确认所有准许上车的新需求是否已完全合入对应分支。
- **风险排查**：存疑修改必须在版本发布前与需求Owner及开发对齐，严防“半拉子”工程（部分合入导致功能异常）。

- **维度2：新配屏、派生与二资源切换适配核对**

- 硬件参数、电声（AQ）参数、画质（PQ）参数是否就绪。
- 软件SID配置是否已更新。
- 电子卡牌资源是否已合入。
- 底层硬件（新屏、DDR、eMMC）的软件适配代码是否齐套。

- **维度3：关键问题与售后修复核对**

- 一级/严重售后客诉问题修复代码是否已合入。
- 其他机芯开发过程中的“举旗（预警）”问题是否需要同步横向展开合入。

- **维度4：评审阻塞（Block）问题释放**

- 所有阻塞版本的Block问题是否全部Release并入库。
- **硬性指标**：系统遗留问题处理率（一般IPR > 86则默认必解，达不到不可发布）。

- **维度5：季度升级与安全合规**

- 季度升级SPL（Security Patch Level）是否已导入。
- 大版本号、内部版本号是否已同步更新。

## **四、 高危风险点管控**

针对极易导致版本Fail的四大高频风险场景，制定专项防范机制：

### 风险点1：APK集成与更新风险

- **风险描述**：APK更新通常只提供集成路径，Commit中可能夹杂大量未评审需求，或静默更改了APK分支/路径，导致构建失败或功能崩溃。
- **管控动作**：

- **深度穿透**：产品SE必须和APKowner核对，了解应用修改的真实范围。
- **路径锁定**：禁止开发人员在随意更改APK集成路径和分支。
- **变更升级机制**：若确需切换APK分支/路径，开发Owner**必须**提前发送正式邮件，说明新旧路径差异、切换原因及潜在风险，经产品SE确认无风险后方可执行。

### 风险点2：资源文件（eSticker等）遗漏合入风险

- **风险描述**：新项目或派生项目初期，常遗漏电子标贴（eSticker）等静态资源，导致打包后返工。
- **管控动作**：

- **前置关卡**：在项目极早期（如B001版本），强制设定校验关卡，必须确认对应PID的eSticker等UI资源已全量合入。

### 风险点3：PQ/AQ与SOC底层联调跨域协同风险

- **风险描述**：PQ/AQ工程师仅上传了DB或Bin文件（如tcon bin），但忽略了与之配对的SOC底层代码/驱动修改。产品SE仅合并了表层参数，导致底层不匹配，出现严重的音画质Bug甚至黑屏。
- **管控动作**：

- **精准校验**：产品SE在按Spec修改DB或者合入DB时，需要和PQ owner二次确认DB路径及PID绑定的正确性。
- **关联强绑**：产品SE在Merge PQ/AQ提交时，必须向相关人员发起询问确认是否依赖SOC底层修改，若依赖，需SOC开发提供对应Code链接并同步入库。

### 风险点4：FCM配置文件OTA失效风险

- 

- **风险描述**：仅修改了配置内容，未更新版本号，导致终端用户OTA升级后，新配置被系统缓存机制忽略，无法生效。
- **SOP管控动作**：
**联动修改规则**：凡修改 tvcust 仓下 system_config.xml，**强制要求**同步修改 component_version 目录下的 system_config_version 值。
- **版本号递增规范**：
新增配置：小数点前数值 +1，小数点后归 0（例：1.5 -> 2.0）。
- 更改既有配置：小数点前不变，小数点后数值 +1（例：1.5 -> 1.6）。

## **五、代码流转与审核机制**

1. **“三分支”强管控原则**：

- 所有代码修改严禁直推主干或量产分支。
2. **流转路径**：开发分支 -> [验证通过] -> 主干分支 -> [合流成功] -> 量产分支 。
3. **强制Review闭环**：
所有提交必须经过对应**模块Owner/领域SE**的Code Review。
4. 未经Review批准的代码，产品SE一律拒绝合入发布版本。

## **六、异常处理与争议决策**

**阻碍合入争议**：若因需求未齐套或Review不通过导致无法合入，但该需求属“紧急商业承诺”，需由业务方发起《风险接受函》，经项目经理(SPM)、质量SE、产品SE联合签字特批后，以Feature Switch（功能开关）的形式隔离合入。
- **拦截与打回**：对于未按规范提供差异邮件说明（如APK路径更换）、未同步依赖代码（如PQ+SOC脱节）的提交，产品SE有权且应当执行**硬性拦截（Reject）**，直到整改完成。
