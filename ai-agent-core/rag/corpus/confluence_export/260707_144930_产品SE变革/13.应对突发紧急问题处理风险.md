# 13.应对突发紧急问题处理风险

| 具体内容 | 遇到紧急问题时怎么组织有效攻关最终将问题解决 |
| 操作方法 | 在项目中往往会遇到一些突发情况，例如工厂生产问题，售后问题等，作为产品SE如何来处理这些紧急问题，如果处理不当或者不及时可能会存在问题扩散或者影响生产的风险。因此需要有一套行之有效的流程或者动作。这里梳理下关键动作：  1.定范围：首先了解详细了解问题情况例如：如何复现，如何恢复，问题概率，使用场景，影响范围等。对于疑难问题还要分析log，初步确认问题范围。也对问题紧急度，严重度有一个初步把握。  2.锁资源：如果是比较明显的问题可以很快确认相关方，如果是疑难问题先从问题发生场景，log相关报错信息，操作涉及到的模块出发，大体确认问题相关方（另可错抓，不可漏网一人），快速建群，锁定资源，存在资源冲突快速上升。这个阶段需要产品SE主导  3.定主责：通过步骤二相关方的深入分析基本上可以确认问题模块，从而确认主责人。主责人人确认后，后续问题主导需要问题大A负责，产品SE关注进展和把控风险。  4.跟进度：问题大A确认后，SPM需要及时跟进相关action落地，建议每天三次对齐进展（可以根据action执行情况而定）。遇到block情况及时拉会对齐。  5.走闭环：问题解决后，需要组织相关人员对修改进行review，确认措施是否有效，可信，不引入新问题。提供patch验证，确认这些都无误后可以走上code流程，最后出版本闭环问题。  6.做复盘：每一次紧急攻关都是对我们质量的一次挑战。一场惊心动魄的战役后需要打扫战场，总结成败。问题再哪里，为什么会发生，怎么去改进需要我们在一次次攻关后去总结，这样才能获取更大的成功。 |
