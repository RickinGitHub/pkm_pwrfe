# 4.1.2 RAT需求可行性评估

> pageId: 249743961 | 导出时间: 2026-07-07T14:50:35.192637

| **标题** | 需求可行性评估 |
| --- | --- |
| 项目阶段 | SR1 |
| 角色 | A |
| 输入 | 《软件需求规格说明书》什么形式？内容？ |
| 沟通对象 | 需求工程师、整机产品开发经理、硬件工程师、SOC、涉及专业工程师 |
| 具体内容 | 主导软件团队进行需求可行性评估，输出《需求可行性评估报告》，包括可行性、工作量、风险 |
| 操作方法 | 1. 需求方按照软工提供的操作指引，使用需求模板提交需求或变更需求，具体阐述需求背景、需求描述、需求价值以及变更原因等操作指引：[https://confluence.tclking.com/](https://confluence.tclking.com/)pages/viewpage.action?pageId=41453651 2.  需求管理经理审核需求信息的完整性，如提交信息不完善，驳回给需求方修正  3.  需求管理经理主导需求预审，各责任人输出以下内容   -  需求管理经理进行需求或变更需求价值初评，并输出需求规格书；  4.  需求管理经理识别主责SDT或承接主体并指派  5.  版本经理初评需求的上线时间、区域以及上线产品线  6.  产品经理初评产品可行性及产品特性、涉及领域或模块  7.  领域SE进行初评技术可行性、预估工作量以及技术风险识   -  SDT PM组织可行性分析：   1.  SDT SE分析技术可行性、工作量和风险  2.  测试负责人进行测试可行性分析   -  产品经理组长详细产品方案详细设计评审：  1.SDT产品经理输出产品方案详细设计并拆解到特性级别 2.组织DIC输出交互GUI方案 3.产品经理拉通需求方、体验、SDT、SE、SDT PM等进行评审，确保产品方案能满足业务诉求 |
| 标准与原则 |  |
| 指导文件 | 《需求可行性评估指引》、《需求可行性评估报告模板》 |
| 适用项目范围 | S,A,B项目 |
| 交付件 | 《需求可行性评估报告》 |
