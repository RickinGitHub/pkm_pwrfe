# 10.OTA部署风险

> pageId: 246760823 | 导出时间: 2026-07-07T14:52:59.596682

| 角色 | A |
| 更新时间 |   |
| 沟通对象 | #### SPM，产品SE, 升级经理，VPM |
| 具体内容 | 我们软件释放后需要进行ota部署，部署前需要对所部署的机型进行ota测试，测试通过后才能部署。 |
| 操作方法 | **OTA 链路测试阶段：**     1.正常流程当我们软件已经完成释放，释放单已经签署（即包括内部释放，google approval）后，才能发起网络升级链路测试申请     2.产品SE提供需要部署ota升级的机型。（产品SE需要有个表格或者文档能够记录目前已经释放的机型，包括pid，释放版本，工厂生产版本等信息），如果是老机型，我们可以从环网上面拿到升级数据。     3.产品SE提起网路升级测试申请 以及SQA 制定网络升级策略评审。     4.在ota升级测试过程中出现的问题需要进行评审，评审通过后方能发起在线oa 流程  **发起OTA部署：**      1.由于OTA部署oa流程是由产品SE在线发起，因此产品SE一定要注意对于测试不通过的机型不能部署ota升级，oa 申请单上面不能体现,一定要与SQA测试机型保持一致。      2.ota oa单发起后一定要跟进各个模块owner签署。签署完成后pdf文档发给升级经理进行部署。     3.对于特殊pid 不部署一定要提醒部署负责人注意     4.ota 部署策略一定要跟升级经理对齐，例如0day方式还是后台升级方式，每天放量多少，多少天后可以大量放量。  **OTA部署后：**     1.关注新版本激活情况     2.监控售后数据，是否有共性严重问题。 |
