# 2.需求导入风险

> pageId: 243464051 | 导出时间: 2026-07-07T14:53:05.106233

| 标题 | 需求导入风险 |
| --- | --- |
| 状态 | 已评审 |
| 角色 | A |
| 更新时间 |   |
| 沟通对象 | BA， 需求开发owner |
| 具体内容 | 重大需求上车是否来带性能&稳定性方面风险以及设计方案方面是否存在合规，安全风险进行识别 |
| 操作方法 | 1.参加需求上车评审会议。  2.对需求交付件重点check ：特别是设计文档，性能&稳定性测试报告有没有劣化项。      2.1>方案是否兼顾老平台      2.2>是否与其他模块有耦合性      2.3>是否有feature隔离      2.4>是否需要长驻后台，运行时占用内存大小。      2.5>需要check 上车要素表里面内容是否全部pass。      2.6>需要考虑DFX设计 （例如：可维护性，可生产性）。  3.如果需求对性能，稳定性有影响会带来劣化要求整改完成后才能上车。     3.1>关于 稳定性和性能的各项弱化都需要澄清，性能组长和稳定性组长参与评审  4.若是系统性需求上车，涉及内存管理，进程管控等功能需要上升部门长进行决策。例如：IOlimit,tgruad等类似需求。  5.需求上车后，需要相关领域FDE输出测试策略，确保NPI阶段不会有重大问题引入。  6.在代码提交的时候一定要拉起需求相关owner 一一确认，所有代码是否全部入库，owner给出上库链接。（特别是owner涉及多个模块合作开发时一定要注意，避免漏合，少合code情况）。  7.针对特批上主干分支的情况需要上升到解决方案中心总监审批  8.针对特批上量产分支的情况需要上升到产品部部门长审批 |
| 标准与原则 | 按软件中心项目管理模板输出 |
