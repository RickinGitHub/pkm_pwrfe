# 2.1.3，静态扫描

> pageId: 245932286 | 导出时间: 2026-07-07T14:53:20.537936

| **标题** | **静态扫描** |
| --- | --- |
| 项目阶段 | SR4 |
| 输入 |  |
| 沟通对象 | 工程效能工程师，CIE工程师，Coverity安全部负责人 |
| 具体内容 | 代码质量管理 |
| 操作方法 | ## **项目扫描执行流程**  1）CIE工程师  负责coverity流水线维护，项目配置，解决coverity扫描过程中出现的异常问题，确保结果的输出  2）产品SE  按SR节点需求启动项目扫描构建和评审  3）Coverity安全部负责人  coverity 和Hades 平台的维护，负责结果分类筛选和自动提单  ## **问题闭环流程**  ### 1）Coverity自动提单  目前只会对S/A问题等级的Coveitry问题进行自动提单，即项目的预发布任务在跑CI Coverity扫描时，对扫描出来的问题在Jira平台自动创建Coverity类型的问题单，并指派给相关负责人。  ### 2）Coverity问题解决  来源一：根据上述项目Coverity自动提单后指派的Jira单去修复问题’  来源二：在工作台提交MR进行代码平台评审过程，修复报告提示的Coverity问题  1、在工作台排查哪些文件有Coveirty问题  2、根据问题等级，安排修复问题的先后顺序  来源三：Coverity报告  Coverity扫描报告：包含所有的Coverity问题（安全问题、质量问题）  Coverity红线报告：包含必解的安全问题（SS）+质量问题（S）  ### 3）Coverity问题回归  1. 开发将问题修复后，无需在JIRA平台将Coverity问题的状态手动进行流程，**系统将会自动识别已修复的问题再进行回归；**  2.开发认为是误报，在JIRA单上处理问题按钮；再点击提交误报按钮，并填写误报相关信息     开发组组长收到问题流转，在JIRA上确认该问题是否为误报，如认为是误报，点击误报确认按钮；若不认为该问题是误报，点击驳回误报按钮。  3.如果是安全红线问题，同质量问题处理  详细操作可查看：[Coverity提单及解决流程](https://confluence.tclking.com/pages/viewpage.action?pageId=135597308) |
| 标准与原则 | <1> 针对可定性的问题，可申请豁免 ;  多项目共性问题，可过滤筛选 |
