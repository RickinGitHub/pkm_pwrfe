# 7.性能/稳定性风险

> pageId: 243464058 | 导出时间: 2026-07-07T14:53:11.558852

| 标题 | 性能/稳定性风险 |
| --- | --- |
| 状态 | 已评审 |
| 角色 | A |
| 更新时间 |   |
| 沟通对象 | VPM，性能/稳定性 SE，PM |
| 具体内容 | ### 识别性能/稳定性风险并制定解决措施，确保性能/稳定性达标。各项指标能够pass |
| 操作方法 | 1.按照项目阶段（一般在M1，M4，V1阶段，其他节点就是按修改点以及时间评估），产品SE给出合适版本进行性能/稳定性测试。       稳定性版本建议从下面几点进行把关（具体视项目情况而定）       1.1：需求全部上车       1.2：无重大功能block问题       1.3：无重大已知稳定性或者性能问题  2.稳定性，性能报告出来后，性能/稳定性SE 确认问题归属模块，责任人。参考[https://docs.qq.com/sheet/DTmJOaG5BSnZ6cnNj?tab=1hj53z](https://docs.qq.com/sheet/DTmJOaG5BSnZ6cnNj?tab=1hj53z)  3.问题划分完成后，项目PM需要重点跟进解决闭环。稳定性，性能都是项目重中之中的问题需要重点关注。  4.当问题收敛65%~80%的时候需要再出版本进行稳定性，性能测试。一者回归问题二者确认新修改不会引出新问题。  5.整个问题处理过程性能/稳定性SE主责，SPM跟进解决情况。组织疑难问题攻关。  6.对于以下稳定性问题需要重点关注，及时组织攻关。        6.1:死机问题        6.2:长期未解决或者未彻底解决的问题        6.3 :中间件进程挂死       6.4: kernel painc 问题  7.针对修改bug，是否需要重新提稳定性性能测试，判断标准如下：      7.1 针对bug修改owner需要确认是否影响稳定性或者性能。另外需要稳定性owner/性能owner 确认影响，若无影响则不提测性能/稳定性测试      7.2：若bug修改存在影响，需要同FDE制定测试策略，是否只测试某个场景即可。  8.针对派生机型若无稳定性，性能修改可以复用基础机型稳定性/性能报告，但需要同VPM，质量SE达成一致。 |
