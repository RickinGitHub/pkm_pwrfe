# 2.3.2 分支代码合入

> pageId: 253024629 | 导出时间: 2026-07-07T14:53:24.530502

| 标题 | **分支代码合入** |
| --- | --- |
| 状态 | 已评审 |
| 角色 | A |
| 更新时间 |   |
| 沟通对象 | 领域SE |
| 具体内容 | 看守主干和量产分支的代码质量 |
| 操作方法 | **主干分支代码提交流程：**  1.  准备提交的代码需要保证上游（特性分支或开发分支）已经提交，且功能测试正常、稳定； 2. 使用Cherty-Pick方式将上游的提交同步到主干，确保与上游分支Changeld一致; 3. 保证自测通过，自测报告包括功能测试和稳定性测试，并且符合编程规范； 4. 启动预编译，编译通过后方可启动审核; 5. 评审人进行评审提出问题，代码提交人针对所提问题进行修改后和评审人员确认无误后，方可进入代码合入阶段； 6. 模块SE对提交负责，审核通过打分+2；  7.产品SE识别修改是否是未评审的需求，submit入库； 8. 合入之后代码owner需要使用合入后的代码再次验证，确保功能正常；     **量产分支代码提交流程：**  1.  准备提交的代码需要保证上游（主干分支）已经提交，且功能测试正常、稳定； 2. 使用Cherty-Pick方式将上游的提交同步到主干，确保与上游分支Changeld一致; 3. 保证自测通过，自测报告包括功能测试和稳定性测试，并且符合编程规范； 4. 启动预编译，编译通过后方可启动审核; 5. 评审人进行评审提出问题，代码提交人针对所提问题进行修改后和评审人员确认无误后，方可进入代码合入阶段； 6. 模块SE对提交负责，审核通过打分+2；  7. 产品SE根据项目进度、修改的风险、修改必要性（是否限期释放问题、是否量产必解问题、售后问题），评估是否入库量产分支；如果和领域SE、质量SE达不成一致，则上升至组长评估；  8. 产品SE确认无误，submit入库； 9. 合入之后提交owner需要使用合入后的版本再次验证，确保功能正常； |
| 标准与原则 | - 产品分支（开发分支、主干分支、量产分支）禁止合入未评审的需求； - 量产分支原则上禁止合入新的功能，只能合入量产必解问题的修改，特殊情况评审通过后需部门长决策； - 主干分支和量产分支只有【产品SE】有submit权限； - 紧急版本和售后版本避免引入比较多的修改，则要求在基础版本上拉hotfix分支合入必解问题； |
| 适用项目阶段 | 所有 |
