# 3.跨软件模块/专业协同

> pageId: 247456755 | 导出时间: 2026-07-07T14:50:34.107178

| **标题** | 跨软件模块/专业协同 |
| --- | --- |
| **状态** | 已完成->评审中 |
| **角色** | A |
| **更新时间** |   |
| **沟通对象** | 各软件模块Owner、模块SE、SEG、产品SE、SPM、SQA，AOE, 硬件，SOC，TQC，系统 |
| **具体内容** | 问题收集-->问题分析-->问题闭环->实战举例->工作流指引 |
| **操作方法** | - **问题收集**      （1）SE需要对齐、收集其它专业人员提报的问题，整理完善问题信息   - **问题分析**** **       SE初步需要确认问题是需求还是BUG。             （1）如果是需求，走模块OWNER需求开发流程。             （2）如果是常规BUG，操作步骤如下：                                1)  SQA初步识别模块OWNER，将问题转交给模块OWNER。                                2）SQA不能识别的模块OWNER，由产品SE主导定位模块OWNER（[模块负责人汇总](https://confluence.tclking.com/pages/viewpage.action?pageId=220400400)）。                                3）模块OWNER，给出问题的分析和解决时间。                                4）SE主导梳理BUG的优先级。SPM跟进BUG的解决进度。                                5）针对疑难BUG，产品SE主导进入问题攻关。             （3）如果是疑难BUG，操作步骤如下：                               1）根据问题模块分类，产品SE主导，SPM拉通参会人员，包含模块OWNER, 模块SE，系统SE, SOC等讨论解决方案                               2）如不能理清解决方案，则上升SEG拉会议[（](https://confluence.tclking.com/pages/viewpage.action?pageId=234111361)[SEG攻关流程](https://confluence.tclking.com/pages/viewpage.action?pageId=234111361)[）](https://confluence.tclking.com/pages/viewpage.action?pageId=234111361)，辅助理清技术方向。                               3）技术方向达成一致意见后，由模块OWNER主导实现修改。   - **问题闭环**  **      **需求：                1） SPM根据模块OWNER给出的开发计划闭环跟进。                2）CIE发布测试版本，SQA测试团环。        常规BUG：                 1） SPM跟进模块OWNER BUG的解决计划。根据BUG解决时间，产品SE确定版本发布。                 2）CIE发布测试版本，SQA测试团环。        疑难问题：                 1）SPM整理会议记录，发布工作事项，通知到各参会人员。                 2）模块owner给出解决计划时间点，SPM跟进模块owner进行问题闭环。                 4）CIE发布测试版本，SQA测试团环。   - **实战举例** 问题来源：                    1）工厂QT抽检发现问题，2、架构调试问题 , 3、工厂老化问题  初步分析：                     1）工厂AOE收集基本的问题信息。                     2）工厂AOE会初步排查是软件问题，还是硬件问题，如果AOE排除是硬件问题，则不走软件问题流直，直接联系硬件处理。                     3）如果是软件问题，AOE会发起问题邀请，一般是邮件跟进，影响走货的紧急问题会拉微信群及时响应                     4.）A0E部门提供问题的LOG  问题解决：                    1）快速建立问题微信群，拉通AOE、硬件，产品SE，软件项目管理、由产品SE快速初步梳理问题，确认问题模块责任人                    2）拉通模块责任人，分析对应的LOG，快速定位问题点。                    3）确定解决问题的方法，模块SE review修改  问题验证：                   1）编译升级包提供给AOE验证，验证通过后发布量产正式版本，  小结：                          1）工厂是打粮食的部门，生产的问题需要第一时间响应，如果存在人力资源上面的冲突，需要向项止管理，部门长第一时间审                请资源，避免影响出货或生产效率。   -
